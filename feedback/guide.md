# 프레임워크 개선 인사이트 가이드

## 목적

프레임워크 자체의 비효율과 개선점을 기록하는 공간이다.

프로젝트 작업에서 배운 교훈은 `memory/knowledge/lessons/`에 Lesson으로 남긴다. 여기(`feedback/`)는 **프레임워크 설계**의 문제점을 기록하는 곳이다.

| 구분 | Lesson | Insight |
|------|--------|---------|
| 대상 | 프로젝트 작업 (코드, 연구, 의사결정) | 프레임워크 설계 (절차, 소통, 구조) |
| 위치 | `projects/{name}/memory/knowledge/lessons/` | `feedback/insights.md` |
| 범위 | 해당 프로젝트 | 프레임워크 전체 (프로젝트 무관) |
| Git | ignored (projects/ 안) | tracked |

---

## 누가 기록하나

모든 에이전트가 기록할 수 있다. 작업 중 프레임워크 비효율을 발견하면 `insights.md`에 행을 추가한다. Manager가 주기적으로 정리하고 상태를 업데이트한다.

---

## 언제 기록하나

다음 상황에서 인사이트를 남겨라:

- **소통 병목** — 응답이 너무 느림, 메시지가 전달 안 됨, 정보가 유실됨
- **절차 비효율** — 불필요한 단계, 빠진 단계, 순서가 비합리적
- **역할 경계 문제** — 누가 해야 할지 모호, 업무가 중복되거나 빠짐
- **도구/자동화 부재** — 수동으로 반복하는 작업이 자동화 가능
- **양식/컨벤션 문제** — 문서 양식이 안 맞음, 컨벤션이 불명확

---

## 기록 방법

`insights.md`의 테이블에 행을 추가한다.

```markdown
| 7 | 2026-02-19 | researcher | 절차 | draft | 온보딩 시 domain context를 너무 늦게 읽어서 초반 판단이 부정확 |
```

필드:
- **#**: 순번 (1부터 순차)
- **날짜**: 발견일 (YYYY-MM-DD)
- **발견자**: 에이전트 역할명
- **카테고리**: `소통` / `절차` / `역할` / `도구` / `양식`
- **상태**: `draft` → `validated` → `implemented` 또는 `rejected`
- **요약**: 한 줄 설명

상세 내용이 필요하면 테이블 아래에 섹션을 추가한다:

```markdown
## INS-007: 온보딩 도메인 컨텍스트 순서

- **발견 상황**: ...
- **현재 동작**: ...
- **제안**: ...
- **영향 범위**: ...
```

---

## 상태 관리

| 상태 | 의미 |
|------|------|
| `draft` | 에이전트가 발견하고 기록. 아직 검증 안 됨 |
| `validated` | Manager 또는 유저가 실제 문제임을 확인 |
| `implemented` | 프레임워크에 반영 완료 |
| `rejected` | 검토 후 개선 불필요로 판단 |

---

## 읽기 시점

이 모듈은 에이전트 온보딩 절차에 포함되지 않는다. 다음 상황에서만 참조한다:

- 프레임워크 개선 작업을 할 때
- 유저가 프레임워크 리뷰를 요청할 때
- 축적된 인사이트를 정리할 때
