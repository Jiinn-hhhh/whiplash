# domains/ — 도메인 특화 정의

에이전트가 특정 분야에서 일할 때 읽는 **추가 컨텍스트**.

---

## 개념

도메인은 프로젝트의 분야를 정의한다. 딥러닝, 게임 개발, 웹 개발 등 각 분야마다 고유한 용어, 원칙, 품질 기준이 있다. 도메인 파일은 이러한 분야별 지식을 에이전트에게 제공한다.

---

## 구조

```
domains/
  {domain-name}/
    context.md              # 도메인 배경, 핵심 개념, 용어, 품질 기준
    {role}.md               # 특정 에이전트를 위한 도메인 특화 지침 (선택)
```

### context.md (필수)
- 모든 에이전트가 읽는다.
- 도메인의 배경, 핵심 개념, 용어 정리, 일반적인 품질 기준을 담는다.
- 이 파일을 통해 에이전트가 도메인의 맥락을 이해한다.

### {role}.md (선택)
- 특정 역할의 에이전트만 읽는다 (예: `researcher.md`는 Researcher만).
- 해당 역할이 이 도메인에서 특별히 주의할 점, 추가 방법론, 평가 기준 등을 담는다.
- 없으면 기본 규칙만으로 작업한다.

---

## 핵심 원칙

### 보충, 교체 아님
도메인 파일은 `agents/common/`의 기본 규칙과 에이전트 `profile.md`의 역할 정의를 **보충**한다. 기존 규칙을 무효화하거나 교체하지 않는다.

### 기본 도메인
도메인이 지정되지 않은 프로젝트는 `general`로 간주한다. `general` 도메인은 별도 파일이 필요 없다 — 기본 규칙만으로 충분하다.

### 경량 유지
도메인 파일은 간결하게 유지한다. 에이전트가 빠르게 읽고 맥락을 잡을 수 있을 정도면 된다. 백과사전이 아니라 지도.

---

## 새 도메인 추가

1. `domains/{domain-name}/` 폴더를 생성한다.
2. `context.md`를 작성한다 — 도메인 배경, 핵심 개념, 용어, 품질 기준.
3. 필요 시 `{role}.md`를 작성한다 — 특정 에이전트의 도메인 특화 지침.
4. 프로젝트의 `project.md`에서 도메인을 지정한다.

---

## 온보딩에서의 위치

에이전트 온보딩 시:
1. ... (공통 규칙, 프로젝트 컨벤션, profile.md, project.md 읽기)
2. `domains/{domain}/context.md` 읽기
3. (해당 시) `domains/{domain}/{role}.md` 읽기
4. ... (지식 지도 읽기)

상세는 `agents/common/project-context.md` 참조.
