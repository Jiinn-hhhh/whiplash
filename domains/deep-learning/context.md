# Domain: Deep Learning

딥러닝 프로젝트에서 모든 에이전트가 공유하는 분야 특성.

---

## 딥러닝 프로젝트의 본질

딥러닝은 **설계→구현** 분야가 아니라 **가설→실험→관찰** 분야다.

- **실험 주도.** 정답을 미리 알 수 없다. "이 아키텍처가 이 데이터에서 작동할까?"는 돌려봐야 안다. 계획이 아니라 실험 루프가 진행을 결정한다.
- **비결정론적 결과.** 같은 코드, 같은 데이터라도 초기화·셔플링·하드웨어 차이로 결과가 달라진다. 단일 실행의 숫자를 절대적으로 신뢰하지 않는다.
- **데이터가 모델보다 중요하다.** 대부분의 성능 향상은 더 나은 아키텍처가 아니라 더 나은 데이터에서 온다. 데이터 품질·양·분포에 대한 이해가 모델 선택보다 선행한다.
- **컴퓨팅이 설계를 제약한다.** 이론적으로 최선인 방법이 현실의 GPU/시간/비용 제약 안에서 불가능할 수 있다. 모든 기술적 결정에는 compute budget이 암묵적 변수다.

---

## 이 분야에서의 품질

일반 소프트웨어의 "버그 없는 코드"와 달리, 딥러닝의 품질은 **실험의 신뢰성**이다.

### 재현성
코드만으로는 부족하다. 같은 결과를 다시 얻으려면 코드 + 데이터 + 환경 + config + 랜덤 시드가 모두 기록되어야 한다. 재현 불가능한 결과는 결과가 아니다.

### 공정한 비교
A가 B보다 좋다고 주장하려면, A와 B가 같은 조건(데이터, compute, 튜닝 노력)에서 비교되어야 한다. 한쪽만 최적화한 비교는 의미가 없다.

### 통계적 유의성
단일 실행의 0.3% 향상은 노이즈일 수 있다. 여러 시드, 여러 분할에서의 결과 분포를 봐야 한다.

### 실패의 기록
"이 방법은 안 됐다"도 지식이다. 왜 안 됐는지의 분석이 다음 실험의 방향을 결정한다. 성공한 실험만 기록하면 팀은 같은 실패를 반복한다.

## 팀 간 경계

딥러닝에서 Researcher와 Developer의 경계는 다른 분야보다 명확해야 한다. 프로토타입 코드와 프로덕션 코드의 요구사항이 근본적으로 다르기 때문이다.

### 원칙
- **Researcher는 검증까지.** "이 방법이 작동한다"를 실험으로 보여주는 것까지가 Researcher의 책임이다. 프로덕션 수준의 코드 품질은 요구하지 않는다.
- **Developer는 프로덕션 전환.** Researcher의 결과를 받아 재현하고, 프로덕션 요구사항(속도, 안정성, 확장성)에 맞게 재구현한다.
- **인계 시 넘겨야 하는 것:** 재현에 필요한 모든 정보(config, 환경, 데이터 조건)와 방법의 한계점. 코드 자체보다 "무엇이 왜 작동하고, 어디서 깨지는지"가 더 중요하다.

### DL에서 이 경계가 특히 중요한 이유
- 학습 코드(실험용)와 서빙 코드(추론용)는 구조가 완전히 다르다.
- 노트북에서 잘 되던 것이 분산 환경에서 깨지는 경우가 흔하다.
- 모델 성능이 인프라 변경(GPU 종류, 배치 크기, 정밀도)에 민감하다.
