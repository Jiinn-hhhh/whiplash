## DL 엔지니어링의 특성

딥러닝 코드는 일반 소프트웨어와 근본적으로 다른 점이 있다.

- **Config가 코드만큼 중요하다.** 같은 코드라도 하이퍼파라미터 하나의 차이로 결과가 완전히 달라진다. Config는 코드와 동등한 수준으로 관리·버전 관리되어야 한다.
- **테스트의 의미가 다르다.** 코드가 "맞게" 돌아가도 모델이 "잘" 작동하는 것과는 별개다. 전통적 소프트웨어 테스트만으로는 DL 시스템의 정확성을 보장할 수 없다.
- **학습은 오래 걸리고 중단되기 쉽다.** 시간 단위에서 주 단위까지. 어디서든 재개할 수 있어야 한다. 중단 = 처음부터 다시는 허용되지 않는다.
- **모델은 블랙박스다.** 코드의 버그는 디버깅할 수 있지만, 모델의 나쁜 성능은 원인을 바로 특정하기 어렵다. 코드 문제인지 데이터 문제인지 하이퍼파라미터 문제인지를 분리하는 구조가 필요하다.

## 핵심 엔지니어링 원칙

### Config와 코드의 분리
하이퍼파라미터, 모델 설정, 학습 설정은 코드 밖에서 관리한다. 실험 재현과 비교가 가능하려면 "무엇을 바꿨는지"가 config 차이로 드러나야 한다.

### 재현성 보장
같은 config + 같은 코드 + 같은 데이터 + 같은 환경 = 같은 결과. 이를 위해 랜덤 시드 고정, 환경 명세, 데이터 버전 관리가 필요하다. "내 컴퓨터에서는 됐다"는 DL에서 특히 위험하다.

### 학습 재개 가능성
체크포인트는 모델 가중치뿐 아니라 옵티마이저 상태, 스케줄러 상태, 현재 에폭/스텝을 포함해야 한다. 장시간 학습에서 중단은 예외가 아니라 일상이다.

### 데이터 파이프라인
학습의 병목은 모델이 아니라 데이터 로딩인 경우가 많다. GPU가 데이터를 기다리는 시간은 순수한 낭비다. 전처리·로딩·증강의 효율이 전체 학습 시간을 좌우한다.

### GPU 효율
GPU 사용률이 낮다면 무언가 잘못된 것이다. 배치 크기, 데이터 로딩, 메모리 관리를 점검한다. 비싼 자원을 놀리는 것은 비용과 시간 모두의 손실이다.

## ML 테스트의 다른 성격

DL 시스템의 테스트는 두 층위로 나뉜다.

### 코드 테스트 (전통적)
- 데이터 전처리 로직이 기대한 대로 작동하는가?
- 모델의 입출력 shape이 맞는가?
- Config 로딩과 체크포인트 저장/로드가 정확한가?

### 모델 행동 테스트 (DL 고유)
- 학습이 진행되면 loss가 감소하는가? (sanity check)
- 알려진 쉬운 예제에서 올바른 출력을 내는가?
- 특정 입력 변화에 대해 출력이 합리적으로 변하는가?

**핵심 구분:** 코드가 맞아도 모델이 나쁠 수 있고, 모델이 좋아도 서빙 코드에서 깨질 수 있다. 두 층위를 명시적으로 분리하여 테스트한다.

## Researcher 프로토타입 인수

DL에서 프로토타입→프로덕션 간극이 특히 큰 이유:
- 실험 코드는 단일 GPU·작은 데이터를 전제한다. 프로덕션은 다중 GPU·전체 데이터.
- 노트북 기반의 대화형 실험은 자동화된 파이프라인과 구조가 다르다.
- 학습 시의 전처리와 추론 시의 전처리가 미묘하게 달라서 성능이 떨어지는 일이 흔하다.

### 인수 시 확인할 것
- **재현:** Researcher의 결과를 동일 조건에서 재현할 수 있는가? 재현이 첫 번째 단계다.
- **한계점:** 어떤 조건에서 잘 되고, 어떤 조건에서 깨지는가? 평균 성능만 보지 않는다.
- **환경 의존성:** 특정 GPU, 라이브러리 버전, CUDA 버전에 의존하는가?
- **학습 vs 서빙의 차이:** 학습 시 사용한 전처리·후처리가 서빙에서도 동일하게 적용되는가?
