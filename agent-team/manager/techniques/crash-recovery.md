# 팀원 장애 복구 (Agent Team 모드)

**대상 행동**: "monitor.sh 없는 환경에서 팀원 장애를 감지하고 복구한다"

Agent Team 모드에서는 monitor.sh가 없으므로 Manager가 직접 장애를 감지하고 복구한다.

---

## 1. 장애 감지

### 신호

| 신호 | 의미 |
|------|------|
| SendMessage 무응답 | 메시지를 보냈으나 합리적 시간 내 응답 없음 |
| TaskList 정체 | in_progress 태스크가 오랫동안 변화 없음 |
| idle 알림 지속 | 팀원이 idle 상태로 오래 머무름 (단, 정상적인 대기일 수 있으므로 주의) |

### 판단 기준

- idle 알림은 정상적인 동작이다. 팀원이 메시지를 보내고 idle로 전환되는 것은 정상 흐름이다.
- 장애 의심 기준: 태스크가 할당된 상태에서 **진행 보고도 없고 응답도 없을 때**.
- 긴 작업(코드 작성, 분석 등) 중일 수 있으므로 성급하게 판단하지 않는다.

### 구체적 시간 기준

- 태스크 할당 후 **10분간** 진행 보고도 응답도 없으면 → 1차 확인 SendMessage
- 1차 확인 후 **5분** 무응답 → 2차 확인
- 2차 확인 후 **5분** 무응답 → 장애로 판정, 재스폰 절차 시작

---

## 2. 복구 절차

### 2.1 재시도 (최대 2회)

```
1. SendMessage(type: "message", recipient: "{role}", content: "상태 확인. 현재 진행 중인 작업이 있나?")
2. 응답 대기
3. 응답 없음 → 한 번 더 시도
4. 2회 무응답 → 재스폰 판단
```

### 2.2 재스폰

팀원이 완전히 응답하지 않을 때:

```
1. 해당 팀원의 미완료 태스크를 TaskList로 확인
2. 미완료 태스크를 unassign (TaskUpdate: owner 제거)
3. Task(spawn)으로 같은 역할의 새 팀원 스폰
4. 새 팀원 온보딩 완료 후 미완료 태스크 재할당
```

### 2.3 중간 결과 복구

재스폰된 팀원에게:
- 이전 팀원이 `workspace/teams/{team}/`에 남긴 중간 결과를 확인하도록 지시
- `memory/{role}/`의 메모도 참조하도록 안내

---

## 3. 에스컬레이션

### 같은 역할 2회 크래시

같은 역할의 팀원이 2회 이상 크래시(재스폰 필요)되면:

```
1. 유저에게 보고 — 역할, 크래시 횟수, 마지막 작업 내용
2. 유저의 판단을 받는다:
   - 재시도
   - 해당 역할 없이 진행
   - 작업 방식 변경
```

### 근본 원인 분석

장애 발생 시 가능한 원인을 분석한다:

| 원인 | 대응 |
|------|------|
| 프롬프트 과부하 | 스폰 프롬프트 단순화, 읽기 파일 수 축소 |
| 태스크 복잡도 | 태스크를 더 작은 단위로 분해 |
| 도구 권한 문제 | spawn 시 mode 설정 확인 |
| 외부 리소스 문제 | 해당 리소스 접근 가능 여부 확인 |

Harness 원칙: 실패는 환경 개선의 기회. 크래시 원인을 분석하여 스폰 프롬프트나 태스크 구조를 개선한다.

---

## 4. 예방

### 중간 결과 저장 지시

모든 팀원에게 스폰 시 다음을 지시한다 (spawn-prompts에 포함됨):
- 긴 작업은 중간 결과를 자기 전용 공간에 파일로 저장
- 진행 상태를 주기적으로 기록

### 태스크 크기 관리

- 하나의 태스크가 너무 크면 팀원이 완료하기 전에 장애가 발생할 가능성이 높아진다.
- 가능한 한 작은 단위로 태스크를 분해한다.

## 5. tmux 모드와의 차이

| 항목 | tmux 모드 | agent-team 모드 |
|------|-----------|----------------|
| 감지 | monitor.sh (30초 폴링) | Manager 수동 (SendMessage + TaskList) |
| 자동 리부팅 | monitor.sh → orchestrator.sh reboot (3회) | Manager → Task(spawn) 재스폰 (2회) |
| 리프레시 | orchestrator.sh refresh (handoff.md) | 재스폰 (중간 결과 파일로 인수인계) |
| 로그 | monitor log + mailbox audit log | 없음 (Manager 메모리에 기록) |

## 6. Manager 자체 장애

Manager가 크래시되면 팀 전체가 리더 없이 남는다. 현재 agent-team 모드에서는 자동 복구 메커니즘이 없다.

**대응**: 유저가 `bash agent-team/boot.sh`로 새 세션을 시작한다. 기존 팀원은 종료되므로 처음부터 재시작이 필요하다.

**예방**: Manager는 주요 결정과 진행 상태를 `memory/manager/`에 파일로 남겨, 재시작 시 맥락을 복구할 수 있게 한다.
