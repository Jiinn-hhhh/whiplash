# 이상 징후 보고

- **대상 행동**: "임계치 위반(경고/위험) 시 즉시 보고", "이상 감지 시 추이와 패턴을 분석하고 원인을 추정"

---

## 1. 이상 유형별 보고 시점

| 유형 | 예시 | 보고 시점 |
|------|------|-----------|
| 임계치 위반 | GPU 사용률 0%, 디스크 95% | 즉시 |
| 서비스 중단 | 학습 프로세스 종료, 서버 무응답 | 즉시 |
| 급격한 변화 | 전회 대비 급격한 수치 변동 | 즉시 |
| 점진적 악화 | 디스크 사용량 꾸준히 증가, 메모리 서서히 증가 | 다음 정기 보고에 포함 |

---

## 2. 즉시 보고 구조

```markdown
# [경고/위험] {항목명} 이상 감지

- **Date**: YYYY-MM-DD HH:MM
- **From**: Monitoring
- **To**: Manager
- **긴급도**: 경고 / 위험

## 수치
- **현재 값**: ...
- **임계치**: ...
- **초과량**: ...

## 추이
- 최근 점검 값 나열 (시간순)
- 언제부터 변화가 시작되었는지

## 원인 추정
- 패턴 분석 기반 추정
- 관련될 수 있는 상황/이벤트
- (처방은 포함하지 않음 — "~일 수 있음", "~가 추정됨" 수준까지만)

## 영향 범위
- 이 이상이 다른 항목/작업에 미칠 수 있는 영향
```

---

## 3. 원인 추정 원칙

- **감지 + 추정까지.** "GPU 사용률이 0%다" (감지) + "프로세스가 데이터 로딩에서 블로킹된 것일 수 있다" (추정).
- **처방은 하지 않는다.** "프로세스를 재시작하라", "batch size를 줄여라" 등은 Manager/팀장 영역이다.
- **추정의 근거를 제시한다.** "GPU 사용률이 주기적으로 0%로 떨어지는 패턴이 데이터 로딩 병목에서 자주 관찰되는 패턴이다" 등.
- **불확실성을 표현한다.** 확정적으로 단언하지 않는다. "~일 수 있음", "~가 추정됨", "~패턴과 유사함".

---

## 4. 보고 후 절차

1. Manager에게 보고한다.
2. Manager의 후속 지시를 대기한다.
3. Manager 또는 팀이 조치한 후, 다음 점검에서 해당 항목을 재점검한다.
4. 조치 결과(개선 여부)를 다음 상태 보고서에 기록한다.
5. 반복적으로 발생하는 이상은 패턴으로 기록하고, 정기 보고에서 추이를 보여준다.
