# 개별 에이전트 정의 양식

새 에이전트를 만들 때 이 양식을 따른다.

## 에이전트 폴더 구조

```
agents/{role}/
  profile.md           # 정의 — 역할, 규칙, 원칙 (무엇을/왜)
  techniques/          # 방법론 — 자연어 절차 (어떻게)
    *.md               # 주제별 기술 문서
  tools/               # 자동화 — 미리 짜둔 코드, 스크립트, 설정 (실행)
```

에이전트 개인 메모는 `memory/{role}/`에 저장한다 (agents/ 밖).

**세 레이어로 분리한다:**

| 레이어 | 내용 | 예시 | 변경 빈도 |
|--------|------|------|-----------|
| `profile.md` | 정의 — 무엇을/왜 | "유저에게 보고한다" | 안정적 (역할이 바뀌지 않는 한) |
| `techniques/` | 방법론 — 자연어 절차 | "보고서 양식으로 작성, 선택지와 추천안 함께 제시" | 자유롭게 개선 가능 |
| `tools/` | 자동화 — 미리 짜둔 코드/스크립트/설정 | Slack webhook, API 호출, 자동화 스크립트 | 필요 시 추가/변경 |

- profile.md → techniques/ → tools/ 순으로 구체성이 높아진다.
- 상위 레이어가 변하지 않아도 하위 레이어는 독립적으로 개선할 수 있다.

## 양식

```markdown
# Agent: [에이전트명]

## 신원
- **이름**:
- **소속 팀**:

## 역할
> 어떤 종류의 일을 하는 사람인지 (1-3문장)

### 해야 하는 것
-

### 하면 안 되는 것
- (안티패턴을 명시적으로 기술)

## 기억
### 배경 지식
- 항상 읽는 파일/폴더 목록

### 장기 기억
- 이 에이전트만의 개인 메모 위치 (memory/{role}/)

## 일하는 방식
### 산출물 형식
- 기본은 common/formats.md. 추가/변형이 있으면 여기 기술.

### 품질 기준
- 잘한 것의 기준
- 못한 것의 기준

### 근거 제시
- 결정의 이유를 어떻게 명시하는지 (Cite LESSON 포함)

## 소통
### 기본 관찰 범위
- 자기 팀 공간 + shared/ (공통)
- 추가로 보는 곳이 있으면 여기

### 보고 대상
- 누구에게 결과를 보고하는지
```

## techniques/ 양식

`techniques/` 폴더에는 주제별로 md 파일을 작성한다. 정해진 양식은 없지만, 다음을 포함해야 한다:

- **대상 행동**: profile.md의 어떤 항목을 구현하는지 (예: "유저에게 보고한다")
- **구체적 절차**: 단계별로 무엇을 하는지
- **도구/파일 경로**: 어떤 도구를 쓰고, 어떤 파일에 쓰는지
- **예시**: 가능하면 구체적인 예시 포함

---

## 작성 가이드

### 역할
- 1-3문장으로 이 에이전트가 하는 일의 핵심을 요약한다.
- "해야 하는 것"과 "하면 안 되는 것"을 명확히 구분하여 경계를 설정한다.
- "하면 안 되는 것"은 MARS 안티패턴 명시 패턴을 따른다: 모호하게 쓰지 말고 구체적 행동을 기술.

### 기억
- **배경 지식**: 이 에이전트가 작업 시작 시 반드시 읽어야 하는 파일을 나열한다.
  - 모든 에이전트 공통: `memory/knowledge/index.md`, `common/README.md`
  - 역할별 추가 항목을 여기에 기술한다.
- **장기 기억**: `memory/{role}/` 폴더를 개인 메모 공간으로 사용한다.

### 품질 기준
- 추상적이지 않게 작성한다. "좋은 코드를 작성한다" (X) → "모든 함수에 입력 검증이 있다" (O).
- 잘한 것과 못한 것 모두 기술하여 판단 기준을 명확히 한다.

### 근거 제시
- 모든 에이전트는 결정 시 이유를 명시해야 한다 (공통 원칙).
- 교훈을 참고했다면 `Cite LESSON-NNN` 형식으로 인용한다.
- 역할별로 추가 근거 제시 방식이 있으면 여기에 기술한다.

### 세 레이어 분리
- `profile.md`에는 **"무엇을/왜"**만 쓴다.
- `techniques/`에는 **"어떻게"**를 자연어 절차로 쓴다.
- `tools/`에는 **미리 짜둔 코드/스크립트/설정**을 둔다.
- 예시:
  - profile.md: "유저에게 진행 상황을 보고한다" (O — 정의)
  - profile.md: "workspace/shared/에 보고서 양식으로 md 파일을 생성한다" (X → techniques/로)
  - techniques/: "보고서 양식으로 작성하고, 선택지와 추천안을 함께 제시한다" (O — 절차)
  - tools/: 보고서 자동 생성 스크립트 등 (O — 코드)
- 상위 레이어가 변하지 않아도 하위 레이어는 독립적으로 개선할 수 있다.
