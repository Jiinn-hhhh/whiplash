# 코드 구현

- **대상 행동**: "프로덕션 코드 구현" (#1), "프로토타입→프로덕션 전환" (#2), "백엔드 네이티브 서브에이전트 / agent team 활용" (#10)

---

## 구현 라이프사이클

```
요구사항 분석 → 모듈 분해 → 구현 → 테스트 → 리뷰 → release handoff
```

---

## 1. 요구사항 분석

- Manager의 작업 지시서에서 "무엇을 만들어야 하는지"를 추출한다.
- Researcher의 제안에서 "왜 이렇게 해야 하는지"를 파악한다.
- 불명확한 요구사항은 Manager 또는 Researcher에게 확인 요청한다.

---

## 2. 모듈 분해

전체 재작성이 아닌 모듈 단위 수정이 기본 원칙이다.

### 절차

1. 변경이 필요한 범위를 식별한다.
2. 독립적으로 구현/테스트 가능한 모듈 단위로 분해한다.
3. 모듈 간 인터페이스(입출력 계약)를 먼저 정의한다.
4. 각 모듈을 백엔드 네이티브 서브에이전트 / agent team에 위임하거나 직접 구현한다.

### 서브에이전트 / agent team 위임 시

- Claude Code든 Codex CLI든 백엔드가 네이티브 팀 기능을 제공하면 우선 활용한다.
- 모듈 단위로 작업을 위임한다. 하나의 서브에이전트가 전체를 보지 않아도 되게 한다.
- 인터페이스 명세, 입출력 예시, 테스트 케이스를 함께 제공한다.
- 서브에이전트 / agent team의 결과를 통합 테스트로 검증한다.

---

## 3. Researcher 프로토타입 인수

Researcher의 프로토타입을 프로덕션으로 전환하는 절차.

### 절차

1. `workspace/shared/`에서 Researcher의 제안/프로토타입을 확인한다.
2. 프로토타입 코드와 실험 결과를 검토한다.
   - 핵심 로직과 검증된 파라미터를 추출한다.
   - 하드코딩, 에러 처리 부재, 확장성 부족 등을 식별한다.
3. 프로덕션 수준으로 재설계한다.
   - 모듈 구조, 에러 처리, 설정 관리, 테스트를 추가한다.
   - 프로토타입을 그대로 복사하지 않는다. 재설계한다.
4. 구현하고, Researcher에게 원래 의도와 다른 점이 없는지 확인 요청한다.

### Researcher ↔ Developer 경계

| Researcher | Developer |
|-----------|-----------|
| 탐색적/검증용 코드 (jupyter, 스크립트) | 프로덕션/애플리케이션 코드 |
| 가설 검증용 프로토타입 | 구현/테스트/인터페이스 정리 |
| "이렇게 하면 좋겠다" (설계 제안) | "이렇게 구현한다" (실행) |

- Researcher가 프로덕션 코드를 직접 작성하지 않는다.
- Developer가 실험/연구 방향을 직접 결정하지 않는다.
- 경계가 모호할 때: "이 코드가 일회성 검증인가, 반복 사용되는 시스템인가?"로 판단한다.

---

## 4. Fail-safe 설계

실패가 치명적이지 않고 복구 가능한 구조로 설계한다.

### 원칙

- 에이전트가 잘못된 코드를 생성해도 시스템 전체가 망가지지 않아야 한다.
- 격리된 실행 환경, 자동 검증 게이트, rollback-friendly 변경 단위를 둔다.
- 에이전트가 실패할 때 사람이 대신 코드를 쓰지 않는다. 환경/도구/제약을 개선한다.

### 실천

1. 모든 변경은 버전 관리된다 (git).
2. release handoff 전 자동 테스트 게이트를 통과해야 한다.
3. runtime apply는 Systems Engineer/Manager 승인 경로를 따른다.
4. 서브에이전트의 코드는 반드시 테스트/리뷰를 거친다.

---

## 5. 구현 체크리스트

코드 구현 완료 시 확인:

- [ ] 요구사항이 모두 반영되었는가?
- [ ] 모듈 간 인터페이스가 명세대로 동작하는가?
- [ ] 테스트가 작성되었는가? (quality-assurance.md 참조)
- [ ] 에러 처리가 적절한가?
- [ ] 기술 결정의 근거가 기록되었는가? (ADR 또는 코드 주석)
