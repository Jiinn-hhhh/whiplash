## 1. 교훈 추출

### 시점

- release handoff 완료 후
- 장애 해결 후 (장애 분석 보고서에서 도출)
- 아키텍처 결정 후 (ADR에서 일반화 가능한 교훈)
- 기술 부채 해결 후
- Researcher 프로토타입 인수 과정에서 반복되는 패턴 발견 시

### 절차

1. 구현 경험에서 다른 작업/프로젝트에도 적용 가능한 인사이트를 식별한다.
2. 기존 `memory/knowledge/lessons/`의 교훈과 의미적 중복 여부를 확인한다.
   - 중복이면: Manager에게 기존 교훈 보강을 제안한다.
   - 새로운 것이면: `common/formats.md`의 교훈(Lesson) 양식으로 작성한다.
3. Manager와의 디스커션을 통해 교훈의 일반성과 정확성을 검증한다.
4. 검증된 교훈을 `memory/knowledge/lessons/`에 기록한다.

### 교훈이 되기 좋은 것

- "이 기술 스택을 선택했더니 이런 장단점이 있었다"
- "프로토타입에서 프로덕션 전환 시 이 부분이 항상 문제였다"
- "이 장애는 이런 구조적 원인 때문이었고, 이렇게 방지할 수 있다"
- "서브에이전트에게 이렇게 지시했더니 더 좋은 결과가 나왔다"

## 2. Cross-branch Transfer

다른 분야/프로젝트의 교훈을 현재 개발에 적용한다.

### 절차

1. 새 구현을 시작할 때 `memory/knowledge/lessons/`를 검토한다.
2. 직접 관련되지 않더라도 구조적으로 유사한 교훈을 탐색한다.
   - 예: 데이터 파이프라인 교훈이 로그 수집 시스템에도 적용 가능한가?
   - 예: Researcher의 실험 교훈이 API/서비스 구조 설계에 시사점을 주는가?
3. 적용 가능한 교훈을 발견하면 현재 구현에 어떻게 적용할지 기술한다.

### 원칙

- 교훈은 발생한 분야에만 갇히지 않는다. 구조적 유사성을 적극적으로 탐색한다.
- 전이 적용 시 원래 맥락과 현재 맥락의 차이를 명시한다.

## 3. 산출물 공유

### 팀 내부 → 공유 공간 이동 절차

1. `workspace/teams/developer/` 내부에서 산출물을 작성하고 정리한다.
2. 확정된 산출물만 공유 대상으로 선정한다.
   - 확정 기준: 구현 완료, 테스트 통과, 리뷰 완료
3. 산출물 유형에 따라 이동한다:

| 산출물 | 이동 위치 |
|--------|-----------|
| 교훈 | `memory/knowledge/lessons/` |
| 기술 설계서 (ADR 등) | `memory/knowledge/docs/` |
| 기술 보고서 (장애 분석 등) | `workspace/shared/` 또는 `memory/knowledge/docs/` |
| 사용자용 보고서/산출물 | `reports/` |
| Systems Engineer 협업 메모 | `workspace/shared/` |

4. 중간 결과, 실험 빌드, 미검증 코드는 `workspace/teams/developer/`에 남긴다.

## 4. 기술 보고서 작성

코드 변경 영향, 장애 분석에서의 애플리케이션 관점 정리 등을 `common/formats.md`의 보고서(Report) 양식으로 작성한다.

### 장애 분석 보고서 추가 항목

```markdown
## 타임라인
- HH:MM — 장애 감지
- HH:MM — 원인 식별
- HH:MM — 조치 완료

## 근본 원인
원인 분석 (귀인 분석 포함)

## 재발 방지
- [ ] 어떤 조치를 취할 것인지
```
