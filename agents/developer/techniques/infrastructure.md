# 개발 환경 및 통합 지원

- **대상 행동**: "코드 품질 관리" (#4), "Systems Engineer와 runtime 검증 협업" (#5)

---

## 1. 개발자 관점 환경 지원

Developer는 애플리케이션 구현에 필요한 **로컬/CI 환경**을 정리하고 유지한다.

### 범위

| 환경 | 내용 |
|--------|------|
| 로컬 개발 | dev server, 의존성, 테스트용 데이터, 시드 |
| CI | 린트, 테스트, 빌드, 정적 검사 |
| 앱 설정 | 환경 변수, 설정 파일, feature flag |
| 개발 도구 | formatter, linter, test runner, type checker |

### 절차

1. 코드 구현에 필요한 dev/test 환경 요구사항을 확인한다.
2. 로컬/CI에서 재현 가능한 형태로 설정을 정리한다.
3. runtime 사실이 필요한 부분은 Systems Engineer에게 확인한다.
4. live 인프라 변경이 필요하면 Manager를 통해 Systems Engineer에게 연결한다.

### 원칙

- 애플리케이션 변경은 로컬/CI에서 재현 가능해야 한다.
- live 환경의 사실 판단은 Systems Engineer가 우선한다.
- Developer는 앱 코드에 필요한 설정과 검증 환경을 유지한다.

---

## 2. 앱 설정 관리

### 환경 분리 원칙

| 환경 | 용도 | 안정성 기대 |
|------|------|-------------|
| 개발(dev) | 개발 및 디버깅 | 불안정 허용 |
| 스테이징(staging) | 배포 전 검증 | 프로덕션과 동일 구성 |
| 프로덕션(prod) | 실제 운영 | 안정성 최우선 |

### 환경 설정 원칙

- 환경별 설정은 코드와 분리한다 (환경 변수, 설정 파일).
- 환경 간 차이를 최소화한다 (동일 이미지, 다른 설정).
- 환경 차이로 인한 런타임 위험은 Systems Engineer와 함께 검증한다.

---

## 3. 기술 이슈 협업

### 이슈 접수 경로

| 출처 | 경로 |
|------|------|
| Systems Engineer 분석 | Systems Engineer → Manager → Developer |
| Monitoring 상태 보고 | Monitoring → Manager → Systems Engineer / Developer |
| Manager 지시 | 유저/Manager로부터의 이슈 보고 |

### 대응 절차

1. 앱 코드 변경이 필요한 이슈인지, runtime 검증이 필요한 이슈인지 분리한다.

| 심각도 | 기준 | 대응 |
|--------|------|------|
| 긴급 | 프로덕션 장애, 데이터 손실 위험 | Systems Engineer와 즉시 협업, Manager에게 알림 |
| 높음 | 팀 블로킹, 핵심 기능 장애 | 당일 대응 |
| 보통 | 성능 저하, 불편 | 우선순위에 따라 대응 |
| 낮음 | 개선 사항 | 백로그에 기록 |

2. 코드 수정이 필요한 부분을 분석하고 해결한다.
3. runtime 영향은 Systems Engineer와 함께 검증한다.
4. 교훈을 추출하여 재발 방지에 반영한다.

## 4. 개발 환경 엔지니어링

에이전트의 물리적 실행 환경도 "환경 엔지니어링"의 대상이다.

### 원칙

- 에이전트(서브에이전트 포함)가 실행할 환경을 표준화한다.
- 환경 문제로 에이전트가 실패하면, 에이전트를 고치는 것이 아니라 환경을 고친다.
- 환경 설정을 문서화하여 재현 가능하게 한다.

### 실천

1. 개발 환경 셋업 스크립트/문서를 유지한다.
2. 의존성 버전을 고정한다 (lock file).
3. 서브에이전트가 일할 때 필요한 도구가 사전 설치되어 있도록 한다.
4. live 인프라 변경은 Systems Engineer와 분리된 절차로 다룬다.
