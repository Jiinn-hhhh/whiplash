# 실험 인프라 및 환경 관리

- **대상 행동**: "실험 인프라 구축" (#4), "기술적 이슈 해결" (#5)

---

## 1. 실험 인프라 구축

Researcher가 실험할 수 있는 환경을 구축하고 유지한다.

### 범위

| 인프라 | 내용 |
|--------|------|
| 컴퓨팅 | 서버, GPU, 컨테이너 환경 |
| 실험 추적 | 실험 로깅, 메트릭 수집, 결과 저장 |
| 데이터 | 데이터 파이프라인, 저장소, 접근 권한 |
| 도구 | 개발 도구, 라이브러리, 의존성 관리 |

### 절차

1. Researcher의 인프라 요청을 확인한다 (`workspace/shared/` 또는 직접 요청).
2. 요구사항을 분석하고, 기존 인프라로 대응 가능한지 판단한다.
3. 신규 구축이 필요하면: 설계 → 구축 → 테스트 → Researcher에게 알림.
4. 대규모 리소스가 필요하면: Manager를 통해 에스컬레이션한다.

### 원칙

- Researcher가 인프라 문제로 블로킹되지 않도록 한다.
- 무엇을 실험할지는 Researcher가 결정한다. Developer는 환경만 제공한다.
- 인프라 변경 시 영향 범위를 기록한다.

## 2. 환경 관리

### 환경 분리

| 환경 | 용도 | 안정성 기대 |
|------|------|-------------|
| 개발(dev) | 개발 및 디버깅 | 불안정 허용 |
| 스테이징(staging) | 배포 전 검증 | 프로덕션과 동일 구성 |
| 프로덕션(prod) | 실제 운영 | 안정성 최우선 |

### 환경 설정 원칙

- 환경별 설정은 코드와 분리한다 (환경 변수, 설정 파일).
- 환경 간 차이를 최소화한다 (동일 이미지, 다른 설정).
- 환경 구성을 코드로 관리한다 (Infrastructure as Code).

## 3. 기술 이슈 대응

### 이슈 접수 경로

| 출처 | 경로 |
|------|------|
| Researcher 모니터링 중 발견 | Researcher monitoring.md → 기술 이슈는 Developer에게 전달 |
| Monitoring 에이전트 상태 보고 | Monitoring → Manager → Developer |
| Manager 지시 | 유저/Manager로부터의 이슈 보고 |

### 대응 절차

1. 이슈를 접수하고 심각도를 분류한다.

| 심각도 | 기준 | 대응 |
|--------|------|------|
| 긴급 | 프로덕션 장애, 데이터 손실 위험 | 즉시 대응, Manager에게 알림 |
| 높음 | 팀 블로킹, 실험 중단 | 당일 대응 |
| 보통 | 성능 저하, 불편 | 우선순위에 따라 대응 |
| 낮음 | 개선 사항 | 백로그에 기록 |

2. 원인을 분석하고 해결한다.
3. 장애 분석 보고서를 작성한다 (`common/formats.md` 보고서 양식 기반).
4. 교훈을 추출하여 재발 방지에 반영한다.

## 4. 환경 엔지니어링

에이전트의 물리적 실행 환경도 "환경 엔지니어링"의 대상이다.

### 원칙

- 에이전트(서브에이전트 포함)가 실행할 환경을 표준화한다.
- 환경 문제로 에이전트가 실패하면, 에이전트를 고치는 것이 아니라 환경을 고친다.
- 환경 설정을 문서화하여 재현 가능하게 한다.

### 실천

1. 개발 환경 셋업 스크립트/문서를 유지한다.
2. 의존성 버전을 고정한다 (lock file).
3. 서브에이전트가 일할 때 필요한 도구가 사전 설치되어 있도록 한다.
