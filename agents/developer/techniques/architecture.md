# 시스템 아키텍처 설계

- **대상 행동**: "시스템 아키텍처 설계 및 기술적 의사결정" (#3)

---

## 1. 기술 선택 절차

### 단계

1. **요구사항 분석**: Manager의 지시 또는 Researcher의 제안에서 기술 요구사항을 추출한다.
2. **후보 평가**: 2-3개 후보를 선정하고, 각 후보의 장단점을 비교한다.
3. **결정 + 근거 기록**: ADR 형식으로 결정과 근거를 기록한다.

### 평가 기준

| 기준 | 질문 |
|------|------|
| 적합성 | 요구사항을 충족하는가? |
| 복잡도 | 팀이 유지보수할 수 있는 수준인가? |
| 생태계 | 문서, 커뮤니티, 에이전트 호환성은? |
| 확장성 | 프로젝트 성장 시 병목이 되지 않는가? |

## 2. ADR (Architecture Decision Record)

아키텍처 결정은 되돌리기 비용이 크다. 모든 중요 결정을 ADR로 기록한다.

### 양식

```markdown
# ADR-NNN: [결정 제목]

- **Date**: YYYY-MM-DD
- **Status**: proposed | accepted | deprecated | superseded
- **Deciders**: [참여자]

## 맥락
어떤 문제/필요에 의해 이 결정이 필요한가

## 결정
무엇을 선택했는가

## 대안
검토했으나 선택하지 않은 옵션과 그 이유

## 결과
이 결정으로 인해 예상되는 영향 (긍정적/부정적)
```

### 기록 위치
- `workspace/teams/developer/decisions/` — 작성 중/검토 중
- 확정 후 `reports/` 또는 `memory/knowledge/docs/`로 이동

## 3. 환경 엔지니어링

에이전트가 일하는 레포 구조, 파일 컨벤션이 코드 자체보다 큰 레버리지를 만든다.

### 원칙

- **구조가 행동을 결정한다**: "더 잘 해"가 아니라 "이 구조 안에서 해"라고 제약을 준다.
- **컨벤션 우선**: 암묵적 규칙보다 명시적 파일 컨벤션. 에이전트는 패턴을 읽고 따른다.
- **디렉토리가 인터페이스**: 폴더 구조 자체가 팀 간 계약이 된다.

### 실천

1. 새 프로젝트/모듈 시작 시 코드 전에 디렉토리 구조를 먼저 설계한다.
2. 파일 이름 컨벤션을 README 또는 AGENTS.md에 명시한다.
3. 서브에이전트가 작업할 때 어디에 무엇을 두는지 혼동 없도록 한다.

---

## 4. 컨텍스트 최소화 설계

에이전트가 읽어야 하는 컨텍스트를 줄이는 코드/문서 구조를 설계한다.

### 원칙

- 모듈은 독립적으로 이해 가능해야 한다. 다른 모듈을 전부 읽지 않아도 작업할 수 있어야 한다.
- 인터페이스(API 계약)와 구현을 분리한다.
- AGENTS.md / README는 지도 수준으로 유지한다 (~100줄).

### 실천

1. 모듈 간 의존성을 최소화하고, 의존 방향을 단방향으로 유지한다.
2. 각 모듈에 짧은 README를 둔다: 목적, 입출력, 의존성.
3. 서브에이전트에게 작업을 위임할 때, 해당 모듈만 읽으면 되도록 한다.

## 5. Harness 설계

같은 모델이라도 구조(harness)에 따라 결과가 2배 이상 차이 난다. 아키텍처가 곧 경쟁력이다.

### 핵심

- 모델을 바꾸는 것보다 모델이 일하는 구조를 바꾸는 것이 더 큰 성능 향상.
- CI/CD, 린팅, 자동 검증 등 자동화된 피드백 장치가 harness의 핵심 구성요소.
- 에이전트가 실패하는 지점을 발견하면 환경/도구/제약을 개선한다.

### 서브에이전트 활용

- Claude Code와 Codex CLI의 네이티브 subagent / agent team 기능을 우선 활용한다.
- 서브에이전트에게 명확한 인터페이스와 제약을 제공한다.
- 작업 범위를 모듈 단위로 한정한다.
- 자동 검증(테스트, 린트)으로 결과를 즉시 피드백한다.
