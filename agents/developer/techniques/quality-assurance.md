# 품질 보증 및 release handoff

- **대상 행동**: "코드 품질 관리" (#4), "release-ready 변경 준비" (#5)

---

## 1. 테스트 전략

### 테스트 피라미드

| 레벨 | 범위 | 속도 | 수량 |
|------|------|------|------|
| Unit | 함수/클래스 단위 | 빠름 | 많음 |
| Integration | 모듈 간 연결 | 중간 | 적당 |
| E2E | 전체 시스템 흐름 | 느림 | 소수 |

### 원칙

- 모든 프로덕션 코드에는 테스트가 동반된다.
- 서브에이전트가 작성한 코드도 테스트 없이 머지하지 않는다.
- 버그 수정 시 해당 버그를 재현하는 테스트를 먼저 작성한다.
- Researcher 프로토타입 인수 시 핵심 로직에 대한 테스트를 추가한다.

---

## 2. 코드 리뷰

### 리뷰 체크리스트

- [ ] 요구사항 충족 여부
- [ ] 테스트 존재 및 커버리지
- [ ] 에러 처리 적절성
- [ ] 보안 취약점 (입력 검증, 인젝션 등)
- [ ] 성능 이슈 (불필요한 연산, 메모리 누수)
- [ ] 코드 가독성 및 컨벤션 준수

### 서브에이전트 코드 리뷰

- 서브에이전트 / agent team의 코드는 Developer(팀장)가 리뷰한다.
- 자동화된 린팅/포매팅 검사를 통과한 후에만 리뷰에 올린다.
- 리뷰 피드백은 구체적으로 작성한다: "이 부분이 문제" + "이렇게 수정" + "그 이유".

---

## 3. CI/CD 파이프라인

피드백 루프를 자동화한다. 일회성 지시보다 자동 검증이 더 강력하다.

### 파이프라인 구성

```
코드 변경 → 린트/포맷 검사 → 단위 테스트 → 통합 테스트 → 빌드 → release handoff
```

### 원칙

- 모든 단계가 자동으로 실행된다. 수동 단계를 최소화한다.
- 어떤 단계에서든 실패하면 즉시 중단하고 피드백을 제공한다.
- 파이프라인 자체도 코드로 관리한다 (버전 관리).
- 서브에이전트의 PR도 동일한 파이프라인을 통과해야 한다.

### Credit Assignment (귀인 분석)

**프로덕션 장애 발생 시에만** 수행한다. 매 변경마다 하지 않는다.
- 문제 발생 시 git bisect 등으로 원인 변경을 식별.
- 결과를 장애 보고서에 기록.

---

## 4. release handoff

### handoff 절차

1. 스테이징 환경에서 검증을 완료한다.
2. 영향 범위와 검증 결과를 정리한다.
3. runtime/deploy 항목은 Systems Engineer에게 전달한다.
4. **고위험 변경**(인증, 데이터 마이그레이션, 인프라)은 Manager 승인 후 배포. 저위험 변경은 Systems Engineer가 바로 배포 가능.

### rollback 입력 정보

- 에러율이 배포 전 대비 유의미하게 증가한 경우.
- 핵심 기능이 동작하지 않는 경우.
- 데이터 정합성에 문제가 발생한 경우.

### 원칙

- 코드 변경은 항상 롤백 가능한 형태로 준비한다.
- runtime 롤백 절차는 Systems Engineer가 소유하므로 필요한 정보를 명확히 넘긴다.
- 외부 반영과 롤백은 사용자 승인 없는 자율 실행 대상이 아니다.

---

## 5. 운영 모니터링

정기적인 인프라/환경 모니터링은 **Monitoring 에이전트**가 담당한다. runtime/live 경로 분석은 **Systems Engineer**가 담당한다. Developer는 다음 상황에서만 직접 모니터링한다:

- **배포 직후 단기 모니터링**: 배포 후 에러율, 성능 지표를 확인하여 롤백 판단
- **장애 대응 중 직접 확인**: Monitoring이 감지한 이상에 대해 Manager 지시로 조치할 때 필요한 상세 진단

Monitoring 에이전트의 상태 보고는 Manager를 통해 Developer에게 전달된다.
