## 1. 테스트 전략

### 테스트 피라미드

| 레벨 | 범위 | 속도 | 수량 |
|------|------|------|------|
| Unit | 함수/클래스 단위 | 빠름 | 많음 |
| Integration | 모듈 간 연결 | 중간 | 적당 |
| E2E | 전체 시스템 흐름 | 느림 | 소수 |

### 원칙

- 모든 프로덕션 코드에는 테스트가 동반된다.
- 서브에이전트가 작성한 코드도 테스트 없이 머지하지 않는다.
- 버그 수정 시 해당 버그를 재현하는 테스트를 먼저 작성한다.
- Researcher 프로토타입 인수 시 핵심 로직에 대한 테스트를 추가한다.

## 2. 코드 리뷰

### 리뷰 체크리스트

- [ ] 요구사항 충족 여부
- [ ] 테스트 존재 및 커버리지
- [ ] 에러 처리 적절성
- [ ] 보안 취약점 (입력 검증, 인젝션 등)
- [ ] 성능 이슈 (불필요한 연산, 메모리 누수)
- [ ] 코드 가독성 및 컨벤션 준수

### 서브에이전트 코드 리뷰

- 서브에이전트의 코드는 Developer(팀장)가 리뷰한다.
- 자동화된 린팅/포매팅 검사를 통과한 후에만 리뷰에 올린다.
- 리뷰 피드백은 구체적으로 작성한다: "이 부분이 문제" + "이렇게 수정" + "그 이유".

## 3. CI/CD 파이프라인

피드백 루프를 자동화한다. 일회성 지시보다 자동 검증이 더 강력하다.

### 파이프라인 구성

```
코드 변경 → 린트/포맷 검사 → 단위 테스트 → 통합 테스트 → 빌드 → 배포
```

### 원칙

- 모든 단계가 자동으로 실행된다. 수동 단계를 최소화한다.
- 어떤 단계에서든 실패하면 즉시 중단하고 피드백을 제공한다.
- 파이프라인 자체도 코드로 관리한다 (버전 관리).
- 서브에이전트의 PR도 동일한 파이프라인을 통과해야 한다.

### Credit Assignment (귀인 분석)

코드 변경이 성능 변화/장애의 원인인지 추적한다.

#### 절차

1. 변경 전후의 지표를 기록한다 (테스트 결과, 성능 메트릭, 에러율).
2. 문제 발생 시 어떤 변경이 원인인지 식별한다.
   - git bisect, 변경 이력 분석, 롤백 테스트 활용.
3. 확실한 귀인과 불확실한 귀인을 구분한다.
4. 결과를 장애 분석 보고서에 기록한다.

## 4. 배포 및 롤백

### 배포 절차

1. 스테이징 환경에서 검증을 완료한다.
2. 배포 계획을 수립한다 (영향 범위, 롤백 계획).
3. 프로덕션에 배포한다.
4. 배포 후 모니터링을 수행한다 (에러율, 성능 지표).

### 롤백 기준

- 에러율이 배포 전 대비 유의미하게 증가한 경우.
- 핵심 기능이 동작하지 않는 경우.
- 데이터 정합성에 문제가 발생한 경우.

### 원칙

- 배포는 항상 롤백 가능해야 한다.
- 롤백 절차는 배포보다 빠르고 단순해야 한다.
- 긴급 롤백은 Developer 자율 판단으로 즉시 수행하고, 사후에 Manager에게 보고한다.

## 5. 운영 모니터링

### 모니터링 대상

| 항목 | 지표 |
|------|------|
| 시스템 상태 | CPU, 메모리, 디스크, 네트워크 |
| 애플리케이션 | 에러율, 응답 시간, 처리량 |
| 인프라 | 서버 상태, 컨테이너 상태, GPU 사용률 |

### 알림 기준

- 임계치 기반 알림을 설정한다.
- 알림 피로를 방지한다: 중요한 알림만 즉시 전달, 나머지는 대시보드.
- Researcher 모니터링(연구 관점)과 Developer 모니터링(인프라 관점)은 분리된다.
