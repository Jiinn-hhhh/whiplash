# 변경 안전성

- **대상 행동**: "변경 전 영향 범위/안전장치/검증 계획 정리" (#4), "로컬 자동화/운영 스크립트 개선" (#5)

---

## 1. 제0 원칙

- 로컬 파일 수정, 테스트, 빌드, 로컬 git commit은 허용한다.
- 원격 시스템 write는 `team/systems-engineer.md`가 허용할 때만 수행한다.
- 문서에 없거나 애매한 변경은 수행하지 않는다. Manager에게 올리고, 사용자 합의 후 문서를 갱신한다.

---

## 2. 변경 전 체크

1. 변경 대상이 local-only인지, remote system change인지 먼저 분리한다.
2. 영향 환경을 명확히 한다.
   - `prod`
   - `staging`
   - `dev/local`
3. `team/systems-engineer.md`의 환경별 권한 표를 확인한다.
4. 롤백 경로와 검증 포인트를 정리한다.
6. 문서에 없거나 애매하면 Manager에게 escalation한다.

---

## 3. 문서 게이트

- `team/systems-engineer.md`는 환경별 권한, 허용된 변경 범위, 실제 시스템 표면을 정의한다.
- 문서에 명시되지 않은 원격 시스템 write는 기본 거부다.

---

## 4. Systems Engineer의 역할

- 문서 안에 있는 범위에서만 실제 반영을 수행한다.
- 반영 전 필요한 정보만 정리한다.
  - 무엇이 바뀌는지
  - 어디에 적용되는지
  - 실패 시 어떤 증상이 나는지
  - 확인해야 할 지표가 무엇인지
- 새로운 표면을 만져야 하면 먼저 문서 갱신 요청안을 만든다.

---

## 5. 권장 산출물

- 변경 영향 보고서
- 정책 갱신 요청 초안
- 사전 체크리스트
- 적용 후 검증 체크리스트
