# 실험

- **대상 행동**: "실험 설계 및 수행"

---

## 실험 라이프사이클

```
가설 설정 → 실험 설계 → 실행 → 결과 분석 → 교훈 추출
```

---

## 1. 가설 설정

- 검증하고자 하는 명제를 명확히 기술한다.
- 좋은 가설: "A를 적용하면 B 지표가 C만큼 개선될 것이다"
- 나쁜 가설: "이게 더 나을 것 같다"

---

## 2. 실험 설계

### 구조

```markdown
# 실험: [제목]

- **Date**: YYYY-MM-DD
- **Hypothesis**: [가설]
- **Method**: [방법 요약]

## 변수
- **독립 변수**: 변경하는 것
- **종속 변수**: 측정하는 것
- **통제 변수**: 고정하는 것

## 베이스라인
기존 결과/방법 (비교 대상)

## 실행 계획
1. 단계별 실행 절차
2. 사용할 도구/환경
3. 예상 소요 시간

## 성공 기준
어떤 결과가 나오면 가설을 지지하는지
```

### 원칙
- 한 번에 하나의 변수만 변경한다 (가능한 경우).
- 베이스라인이 없는 실험은 의미 없다.
- 실행 전에 성공 기준을 정한다 (사후 해석 편향 방지).

---

## 3. 실행

- 프로토타이핑/검증 수준의 코드를 작성한다 (jupyter, 스크립트).
- Claude Code와 Codex CLI의 네이티브 subagent / agent team 기능을 우선 활용하여 실행한다.
- 실행 로그를 기록한다 (`workspace/teams/research/experiments/`).

### Developer와의 경계
| Researcher | Developer |
|-----------|-----------|
| 탐색적/검증용 코드 (jupyter, 스크립트) | 프로덕션/시스템 수준 코드 |
| 가설 검증용 프로토타입 | 실험 인프라 구축 |
| "이렇게 하면 좋겠다" (설계 제안) | "이렇게 구현한다" (실행) |

- 프로덕션 코드/인프라가 필요하면 Developer에게 요청한다.

---

## 4. 결과 분석

### Credit Assignment (귀인 분석)
결과를 분석할 때 **"어떤 변경이 성능 변화의 원인인지"** 귀인 분석을 수행한다.

#### 절차
1. 실험 전후의 차이를 정량적으로 기록한다.
2. 변경한 요소 중 어떤 것이 결과에 기여했는지 분석한다.
3. 확실한 귀인과 불확실한 귀인을 구분한다.
4. 불확실한 경우 추가 실험(ablation)을 고려한다.

### 결과 기록 구조

```markdown
## 결과
- 정량 결과: [수치/지표]
- 정성 결과: [관찰]

## 귀인 분석
- 확실: [어떤 변경이 어떤 효과를 냈는지]
- 불확실: [추가 검증 필요한 부분]

## 결론
가설 지지 여부와 근거
```

---

## 5. 실패 기록

실패한 실험도 반드시 기록한다.

```markdown
## 실패 기록
- **무엇이 안 되었나**: [결과]
- **왜 안 되었나**: [분석]
- **교훈**: [다음에 피해야 할 것 / 시도할 것]
```

- 실패를 숨기거나 생략하지 않는다.
- 실패에서 도출된 교훈이 성공보다 가치 있을 수 있다.
