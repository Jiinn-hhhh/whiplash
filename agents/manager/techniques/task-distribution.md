# 작업 분배

- **대상 행동**: "유저 목표를 팀 단위 작업으로 분해하여 팀장에게 지시"

---

## 절차

1. **목표 분석**: 유저의 목표를 파악하고 필요한 팀/역량을 식별한다.
2. **작업 분해**: 팀 단위로 독립 실행 가능한 작업으로 나눈다. 가능한 병렬 진행을 우선한다.
3. **지시서 작성**: 아래 구조로 작업 지시서를 작성한다.
4. **게시**: `workspace/shared/announcements/`에 md 파일로 게시한다.

---

## 작업 지시서 구조

```markdown
# [작업 제목]

- **Date**: YYYY-MM-DD
- **From**: Manager
- **Priority**: high | medium | low

## 목표
> 이 작업이 달성해야 하는 것 (1-2문장)

## 배경
왜 이 작업이 필요한지, 유저의 원래 요청이 무엇인지

## 팀별 할당

### [팀 A]
- 할 일: ...
- 산출물: ...

### [팀 B]
- 할 일: ...
- 산출물: ...

## 의존성
- [팀 B]는 [팀 A]의 X 산출물이 필요 (해당 시)

## 마감/우선순위
- 기한 또는 우선순위 정보
```

---

## 원칙

- **팀장에게 지시한다**: 개별 팀원이 아닌 팀장에게 전달한다.
- **팀장 내부 fan-out은 맡긴다**: Manager는 목표, 제약, 우선순위, 산출물만 정한다. 어떤 specialist subagent를 호출할지는 각 팀장이 결정한다.
- **병렬 우선**: 가능하면 여러 팀이 동시에 진행할 수 있도록 의존성을 최소화한다.
- **배경을 충분히 제공한다**: 팀장이 왜 이 작업을 하는지 이해할 수 있어야 한다.
- **산출물을 명확히 한다**: 각 팀이 무엇을 내놓아야 하는지 구체적으로 기술한다.

---

## 역할별 기본 배정 힌트

- `Systems Engineer`: live 구조 파악, AWS/클라우드, 배포 경로, 런타임 드리프트, 변경 영향 분석
- `Developer`: 앱 코드 구현, 테스트, 인터페이스 변경, 제품 기능 완성
- `Researcher`: 외부 자료 조사, 비교 분석, 실험/검증
- `Monitoring`: 정기 점검, 상태 감지, 경고/위험 알림

운영/배포/런타임 관련 태스크는 가능하면 Systems Engineer가 먼저 현재 상태를 검증하고, 그 결과를 바탕으로 Developer 태스크를 분리한다.
