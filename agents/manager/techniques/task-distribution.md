# 작업 분배

- **대상 행동**: "유저 목표를 팀 단위 작업으로 분해하여 팀장에게 지시"

---

## 절차

1. **목표 분석**: 유저의 목표를 파악하고 필요한 팀/역량을 식별한다.
2. **작업 분해**: 팀 단위로 독립 실행 가능한 작업으로 나눈다. 가능한 병렬 진행을 우선한다.
3. **지시서 작성**: 아래 구조로 작업 지시서를 작성한다.
4. **게시**: `workspace/shared/announcements/`에 md 파일로 게시한다.

---

## 작업 지시서 구조

```markdown
# [작업 제목]

- **Date**: YYYY-MM-DD
- **From**: Manager
- **Priority**: high | medium | low

## 목표
> 이 작업이 달성해야 하는 것 (1-2문장)

## 배경
왜 이 작업이 필요한지, 유저의 원래 요청이 무엇인지

## 팀별 할당

### [팀 A]
- 할 일: ...
- 산출물: ...

### [팀 B]
- 할 일: ...
- 산출물: ...

## 의존성
- [팀 B]는 [팀 A]의 X 산출물이 필요 (해당 시)

## 마감/우선순위
- 기한 또는 우선순위 정보
```

---

## 원칙

- **팀장에게 지시한다**: 개별 팀원이 아닌 팀장에게 전달한다.
- **병렬 우선**: 가능하면 여러 팀이 동시에 진행할 수 있도록 의존성을 최소화한다.
- **배경을 충분히 제공한다**: 팀장이 왜 이 작업을 하는지 이해할 수 있어야 한다.
- **산출물을 명확히 한다**: 각 팀이 무엇을 내놓아야 하는지 구체적으로 기술한다.

---

## 모니터링 운영

Monitoring 에이전트는 독립 관찰자로, Manager가 직접 운영한다.

### 초기 설정 (온보딩 인계 직후)

1. `project.md`의 모니터링 요구사항을 확인한다.
2. Monitoring 에이전트에게 점검 대상과 주기를 지시한다.
3. Monitoring이 `memory/monitoring/targets.md`를 초기화한다.
4. Manager가 targets.md 내용을 검토하고 승인한다.
5. 승인 후 정기 점검을 시작한다.

### 보고 처리 (런타임)

Monitoring의 상태 보고를 받으면 아래 기준으로 처리한다:

| 상태 | Manager 행동 |
|------|-------------|
| 정상 | 참고만. 별도 조치 불필요. |
| 주의 | 추이를 관찰한다. 악화 가능성이 있으면 예방 조치를 검토한다. |
| 경고 | 영향 범위를 파악하고, 해당 팀장(Developer 등)에게 즉시 조치를 지시한다. |
| 위험 | 즉시 조치 지시 + 필요 시 진행 중인 작업 우선순위를 조정한다. 유저 에스컬레이션이 필요한지 판단한다. |

### 이상 징후 대응 흐름

1. Monitoring이 이상을 감지하고 원인 추정과 함께 보고한다.
2. Manager가 영향 범위를 파악한다 (어떤 작업/팀에 영향이 있는가).
3. 해당 팀장에게 조치를 지시한다 (인프라 문제 → Developer, 실험 관련 → Researcher).
4. 필요 시 진행 중인 작업의 우선순위를 조정한다.
5. 조치 후 Monitoring에게 재점검을 지시하여 결과를 확인한다.
6. 유저에게 알려야 할 수준이면 에스컬레이션한다.

### 주기 조정

프로젝트 상황에 따라 점검 주기를 변경할 수 있다:
- **주기 단축**: 중요 학습 진행 중, 이상 징후 발견 후 추이 관찰 필요 시
- **주기 연장**: 안정적 상태가 지속될 때, 프로젝트 유휴기
- 변경 시 Monitoring에게 새 주기를 지시하고 targets.md에 기록하게 한다.
